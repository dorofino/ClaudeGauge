# Architecture

## Overview

The firmware follows a **state-machine architecture** with three device modes and a cooperative main loop. All rendering uses double-buffered sprites in PSRAM for flicker-free display updates. The firmware supports multiple boards via conditional compilation — see [HARDWARE.md](HARDWARE.md) for board-specific details.

```
┌──────────────────────────────────────────────────┐
│                    main.cpp                       │
│         State machine + main loop                 │
├──────────┬───────────┬───────────┬───────────────┤
│ WiFi     │ Time      │ API       │ Claude.ai │ Web Server│
│ Manager  │ Manager   │ Client    │ Client    │ (Config)  │
├──────────┴───────────┴───────────┴───────────────┤
│    Button Handler    │    Touch Handler (*)       │
├──────────────────────────────────────────────────┤
│  UI Renderer → UI Widgets → Display Manager      │
├──────────────────────────────────────────────────┤
│  Settings Manager (NVS)    │    Data Models       │
└──────────────────────────────────────────────────┘
  (*) Touch Handler only compiled when HAS_TOUCH=1
```

## Device Modes

Defined in `src/main.cpp`:

### MODE_SETUP
Entered on first boot or when WiFi credentials are missing. The device:
1. Creates a WiFi access point (`ClaudeGauge-XXXX`)
2. Starts a DNS server redirecting all domains to the setup portal
3. Serves a web configuration form on port 80
4. Waits for the user to submit WiFi + API key credentials
5. Reboots after saving

### MODE_CONNECTING
Transitional mode on boot when credentials exist:
1. Connects to the saved WiFi network (15s timeout)
2. Syncs NTP time
3. Performs initial API data fetch
4. Falls back to `MODE_SETUP` if WiFi fails

### MODE_DASHBOARD
Normal operation after successful connection:
1. Displays one of 6 dashboard screens
2. Handles button navigation
3. Auto-refreshes data every 5 minutes
4. Auto-dims backlight after 2 minutes
5. Handles WiFi reconnection in the background
6. Continues serving the web portal for reconfiguration

## Main Loop

The `loop()` function runs cooperatively (no RTOS tasks):

```
loop() {
    webServer.handleClient()         // Always process HTTP requests
    check reboot flag                // Reboot if config changed
    buttonHandler.update()           // Poll button GPIOs

    if MODE_SETUP:
        handle backlight dim
        return

    if MODE_DASHBOARD:
        handleNavigation()           // Button → screen change
        handleAutoRefresh()          // Timer → fetch data
        handleBacklightDim()         // Inactivity → dim
        WiFi reconnect check
}
```

The loop runs with a 10ms delay per iteration.

## Component Details

### AppState (`include/data_models.h`)

Central data structure holding all runtime state:

```
AppState
├── current_screen      (ScreenID enum, 0–6)
├── usage               (UsageData — today's token usage)
├── monthly_usage       (UsageData — month-to-date)
├── cost                (CostData — today + month USD)
├── code                (ClaudeCodeData — today)
├── monthly_code        (ClaudeCodeData — month-to-date)
├── wifi_connected, wifi_rssi, wifi_ip
├── api_error, last_error
├── last_refresh, next_refresh, uptime_start
├── claude_ai           (ClaudeAiUsage — subscription rate limits)
└── backlight_level, last_activity, is_fetching
```

### ApiClient (`src/api_client.cpp`)

Makes HTTPS GET requests to `api.anthropic.com:443` using `WiFiClientSecure`. Calls three endpoint categories:

| Method | Endpoint | Data |
|--------|----------|------|
| `fetchUsageReport()` | `/v1/organizations/usage_report/messages` | Token counts by model, grouped by day |
| `fetchCostReport()` | `/v1/organizations/usage_report/cost` | USD cost for a time range |
| `fetchClaudeCodeReport()` | Claude Code analytics endpoint | Sessions, commits, PRs, lines, acceptance rates |

Each fetch uses time ranges generated by `TimeManager`:
- **Today:** `todayStartUTC()` → `todayEndUTC()`
- **Month:** `monthStartUTC()` → `todayEndUTC()`

Authentication is via the `X-API-Key` header with API version `2023-06-01`.

### ClaudeAiClient (`src/claude_ai_client.cpp`)

Fetches Claude.ai subscription rate limits via a cloud proxy (ESP32 cannot connect directly to claude.ai — Cloudflare blocks the mbedTLS TLS fingerprint). The proxy is a Vercel Edge Function deployed from `cloud-proxy/`.

| Method | Proxy Endpoint | Data |
|--------|----------------|------|
| `fetchOrganization()` | `/api/organizations` | Organization UUID |
| `fetchUsage()` | `/api/organizations/{uuid}/usage` | 5-hour and 7-day rate limits, per-model limits |

The client:
- Authenticates via `X-Session-Key` header (proxy forwards as cookie to claude.ai)
- Caches the organization UUID after first fetch
- Parses ISO 8601 reset timestamps to Unix epoch for countdown timers
- Handles session expiry (401), rate limiting (429), and proxy errors (502)

Data is stored in `ClaudeAiUsage` (defined in `data_models.h`) containing `ClaudeAiLimit` structs for each rate window, plus optional extra-usage billing info.

### Data Flow

```
NTP sync → TimeManager generates date strings
                ↓
ApiClient.fetchUsageReport(todayStart, todayEnd) → UsageData
ApiClient.fetchCostReport(todayStart, todayEnd)  → CostData.today_usd
ApiClient.fetchCostReport(monthStart, todayEnd)  → CostData.month_usd
ApiClient.fetchClaudeCodeReport(todayDate)       → ClaudeCodeData
ApiClient.fetchUsageReport(monthStart, todayEnd) → monthly UsageData
ApiClient.fetchClaudeCodeReport(monthStartDate)  → monthly ClaudeCodeData
ClaudeAiClient.fetchUsage()                      → ClaudeAiUsage
                ↓
        All stored in AppState
                ↓
    UIRenderer draws current screen from AppState
                ↓
    DisplayManager pushes sprite to LCD
```

### Display Pipeline

1. **DisplayManager** initializes the TFT and creates a full-screen sprite (`SCR_WIDTH`×`SCR_HEIGHT`, 16-bit) in PSRAM
2. **UIRenderer** draws to the sprite buffer using **UIWidgets** for reusable components
3. **DisplayManager.pushSprite()** transfers the sprite to the LCD in one operation (no flicker)

Screen dimensions (`SCR_WIDTH`, `SCR_HEIGHT`) come from PlatformIO build flags — 320×170 for T-Display-S3, 320×172 for Waveshare 1.47. All layout calculations derive from `SCR_W`/`SCR_H` macros in `ui_widgets.h`.

The sprite is re-rendered every 1 second (for countdown timer) and on any screen change or data refresh.

### Settings Manager (`src/settings_manager.cpp`)

Persists configuration in ESP32 NVS (Non-Volatile Storage):

| NVS Namespace | Key | Value |
|---------------|-----|-------|
| `claudemon` | `wifi_ssid` | WiFi network name |
| `claudemon` | `wifi_pass` | WiFi password |
| `claudemon` | `api_key` | Anthropic API key |
| `claudemon` | `session_key` | Claude.ai session key |
| `claudemon` | `proxy_url` | Cloud proxy URL for Claude.ai requests |

### WiFi Manager (`src/wifi_manager.cpp`)

- Connects with a 15-second timeout
- Auto-reconnects every 30 seconds when disconnected
- Reports RSSI and IP address for the status screen

### Time Manager (`src/time_manager.cpp`)

- Syncs to NTP using three servers: `pool.ntp.org`, `time.google.com`, `time.nist.gov`
- Re-syncs every hour
- Generates ISO 8601 timestamps for API queries
- Generates date-only strings for Claude Code API

### Button Handler (`src/button_handler.cpp`)

Provides a **semantic API** that abstracts physical button differences between boards:

| Method | Two-button boards | Single-button boards |
|--------|-------------------|----------------------|
| `isPrevPressed()` | Left button short press | Always returns `false` |
| `isNextPressed()` | Right button short press | BOOT button short press |
| `isRefreshPressed()` | Right button long press | BOOT button long press |
| `anyPressed()` | Any button event | Any button event |

- 50ms debounce on all buttons
- Detects short press (<800ms) and long press (≥800ms)
- Long press fires once per hold (prevents repeat triggers)
- Uses `#if HAS_TWO_BUTTONS` to compile for dual or single button hardware

### Touch Handler (`src/touch_handler.cpp`)

Only compiled when `HAS_TOUCH=1` (T-Display-S3). Provides capacitive touch via the CST816S controller over I2C:

- `isTouched()` — any touch event (used for backlight wake)
- `isDoubleTapped()` — double-tap detection (400ms window) for force refresh

All `touchHandler` references in `main.cpp` are wrapped in `#if HAS_TOUCH`.

### Web Server (`src/web_server.cpp`)

Dual-purpose HTTP server:

**AP Mode (Setup):**
- Creates `ClaudeGauge-XXXX` access point (last 4 hex of MAC)
- DNS captive portal redirects all domains to `192.168.4.1`
- Serves dark-themed configuration form

**Station Mode (Dashboard):**
- Continues running on the device's local IP
- Allows reconfiguration without factory reset

**Endpoints:**

| Route | Method | Function |
|-------|--------|----------|
| `/` | GET | Configuration form with current status |
| `/save-wifi` | POST | Save WiFi SSID and password, trigger reboot |
| `/save-apikey` | POST | Save API key, trigger reboot |
| `/save-session` | POST | Save Claude.ai session key and proxy URL, trigger reboot |
| `/status` | GET | JSON status (configured, connected, IP, RSSI, heap) |
| `/reset` | POST | Clear all NVS settings, trigger reboot |

## Font System

Fonts are pre-compiled from `Antonio.ttf` into VLW (Variable Length Width) format stored as `PROGMEM` arrays in header files:

| Font | Size | Usage |
|------|------|-------|
| `LCARS_SM` / `SmoothFont12` | 12px | Labels, small text |
| `LCARS_MD` / `SmoothFont18` | 18px | Values, section headers |
| `LCARS_LG` / `SmoothFont28` | 28px | Monthly cost |
| `LCARS_XL` / `SmoothFont36` | 36px | Today's cost (hero number) |

The VLW format includes per-glyph alpha bitmaps for anti-aliased rendering via TFT_eSPI's smooth font support.

## Conditional Compilation

Board-specific behavior is controlled by feature flags defined in `include/pin_config.h`, set based on the board macro (`BOARD_TDISPLAY_S3` or `BOARD_WAVESHARE_147`) from PlatformIO build flags:

| Flag | Effect |
|------|--------|
| `HAS_TWO_BUTTONS` | Enables prev/next navigation vs next-only |
| `HAS_POWER_PIN` | Controls display power enable on init |
| `HAS_TOUCH` | Compiles CST816S touch handler |

Files with conditional compilation:

| File | What's conditional |
|------|-------------------|
| `pin_config.h` | Pin assignments, feature flags |
| `display_manager.cpp` | Power pin initialization |
| `button_handler.h/.cpp` | Single vs dual button state |
| `touch_handler.h/.cpp` | Entire class (excluded when `HAS_TOUCH=0`) |
| `main.cpp` | Touch handler include, init, and usage |
| `ui_renderer.cpp` | Board name on status screen |

All other source files are fully board-agnostic.
